# t4-i2s Design

## Purpose

`t4-i2s` is a LilyGo T4 AMOLED ESP32-S3 application for capturing audio from an
I2S MEMS microphone, visualising the live signal, and recording mono WAV files
to the SD card without blocking audio capture.

The application currently focuses on:

- Live I2S microphone capture.
- FFT, spectrum analyser, and oscilloscope displays.
- Asynchronous SD-card WAV recording.
- WiFi connection and FTP access for downloading recorded WAV files.
- Serial diagnostics for timing, SD-write health, and display load.
- LVGL startup status screen before the live audio displays take over.

## Hardware

- Board: LilyGo T4 AMOLED ESP32-S3.
- Display: 600 x 450 AMOLED, landscape.
- Microphone: Adafruit SPH0645LM4H I2S MEMS microphone breakout.
- SD card: FAT32, tested with a 16 GB card.

Pin assignments:

- I2S BCLK: GPIO 39, defined in code as `kI2SBclk_GPIO`.
- I2S LRCLK / word select: GPIO 40, defined in code as `kI2SLrclk_GPIO`.
- I2S data input: GPIO 41, defined in code as `kI2SData_GPIO`.
- Top button: GPIO 48, defined in code as `kTopButton_GPIO`, normally open,
  active-low.
- Side button: GPIO 21, defined in code as `kSideButton_GPIO`, normally open,
  active-low.

Pin-number constants use a `_GPIO` suffix so it is clear they are physical ESP32
GPIO numbers rather than logical channels or display coordinates.

The Adafruit microphone `SEL` pin is not controlled by firmware. It is pulled to
ground on the hardware so the microphone transmits on its left-channel slot.

## Startup

`amoled.begin()` is intentionally the first function call in `setup()`.

Startup sequence:

1. Initialise AMOLED display.
2. Initialise a simple LVGL startup screen.
3. Start serial and wait for the development serial startup delay.
4. Initialise buttons, colour palette, and Hann window.
5. Allocate the PSRAM-backed recording buffer and FreeRTOS queues.
6. List SD-card files and sizes to serial.
7. Connect to a known WiFi network using credentials from `mercator_secrets.c`.
8. Start the FTP server if WiFi connects.
9. Initialise FFT and I2S.
10. Clear the LVGL startup screen and draw the initial analyser display.
11. Start display, capture, and SD-writer FreeRTOS tasks.

Recording is not started automatically at boot.

The LVGL screen is currently a startup-only status surface. It shows progress
through SD setup, WiFi scans, WiFi connection attempts, and FTP startup. If no
known WiFi is available, the screen explicitly reports that FTP is skipped and
that audio capture will continue offline. Once the analyser display starts,
LVGL is no longer pumped and the existing direct `amoled.pushColors()` rendering
owns the screen.

## Audio Capture

Audio capture is handled by the `capture_fft` task pinned to core 0.

Capture settings:

- Sample rate: 16 kHz.
- FFT frame size: 1024 samples.
- Frame period: about 64 ms.
- WAV format: mono, 16-bit PCM, 16 kHz.
- I2S peripheral: `I2S_NUM_0`.
- I2S read frame count: 128 stereo frames per I2S read.
- Microphone slot: slot 0.

The ESP32-S3 is the I2S master. BCLK and LRCLK/WS are generated by the ESP32 and
sent to the microphone; DOUT is generated by the microphone and read by the
ESP32.

Each capture frame is centred by subtracting the frame mean, windowed with a
Hann window, then passed to the FFT. The original PCM frame is also used for the
oscilloscope and, when recording is active, copied into the recording queue.

## Display

LVGL is initialised during startup through the LilyGo AMOLED LVGL helper. The
helper flushes through the same AMOLED `pushColors()` path as the rest of the
application and uses a PSRAM draw buffer. For now LVGL is deliberately limited
to boot status so it does not compete with the live analyser display task.

The display task is pinned to core 1 and updates three 200 x 135 graphics:

- Spectrum analyser above the FFT display.
- FFT waterfall in the centre.
- Oscilloscope below the FFT display.

The analyser uses 32 configurable bands over roughly 50 Hz to 4 kHz. The FFT
waterfall displays bins 4 to 256, which corresponds to roughly 62.5 Hz to 4 kHz
at a 16 kHz sample rate. The oscilloscope uses the latest PCM frame and applies
automatic vertical scaling.

The FFT waterfall and spectrum analyser use a fixed display noise floor. The
floor is calibrated from the first 5 seconds of FFT frames after the live
display starts, then held constant. This keeps sustained sounds visible instead
of slowly treating them as ambient background. For best visual scaling, keep the
microphone in representative ambient conditions during the first few seconds
after startup.

The side button can request another display-noise calibration. The request is
triggered on button release, then delayed by 500 ms so the physical button click
is not captured as part of the calibration sample. A white `Calibrating` label is
shown in the lower-left corner while calibration is active.

The display frame border shows operating state:

- Green: idle/live capture active.
- Red: recording.
- Yellow: recording start/stop in progress.
- Red error state: I2S or FFT not ready.

If WiFi connects during startup, the SSID and IP address are drawn once in the
lower-right area beside the oscilloscope. The label uses TFT_eSPI's built-in
Font 2 bitmap data but is rendered into a small local pixel buffer before being
sent to the AMOLED. It is static because the WiFi identity is not expected to
change after startup.

Display updates are driven by notifications from the capture task. If several
notifications arrive before the display task catches up, they are coalesced and
reported as display drops. These are display-frame drops, not audio drops.

## SD Recording

Recording is handled by a separate `sd_writer` task pinned to core 0.

The capture task never writes directly to SD. Instead, it copies each 1024-sample
PCM frame into a preallocated ring of recording frames and queues the frame index
for the SD writer.

Recording buffer:

- Frame size: 1024 samples x 2 bytes = 2048 bytes.
- Frame count: 256.
- Approximate audio buffering: 256 x 64 ms = about 16 seconds.
- Allocation preference: PSRAM, with heap fallback.

Recording flow:

1. Top button requests start.
2. SD writer opens the next available `/<n>.wav`.
3. A placeholder 44-byte WAV header is written.
4. PCM chunks are appended from the filled-frame queue.
5. Header is rewritten with the final data size when recording stops.
6. File is flushed and closed before returning to idle.

The recorder periodically flushes the file every 32 chunks. The user should stop
recording and wait for `sd status: idle` before power-off or firmware upload.

## Controls

- Top button: toggle recording start/stop.
- Side button: release to recalibrate the FFT/analyser display noise floor after
  a 500 ms quiet delay.

Button polling is done from the Arduino `loop()` on a non-blocking 10 ms
housekeeping cadence.

## WiFi and FTP

WiFi credentials are kept in `src/mercator_secrets.c`. That file is intentionally
not documented here and should not be read by tooling. The template file shows
the expected variable names and types.

FTP is provided by `SimpleFTPServer` using ESP32 WiFi and SD storage. The FTP
server is configured for:

- Plain FTP.
- No TLS.
- One connection.
- Username/password set in the FTP setup code.

`ftpServer.handleFTP()` runs on every Arduino `loop()` pass. This is important:
the FTP control channel needs frequent polling, otherwise clients can connect
but time out waiting for the welcome message.

## Diagnostics

Serial diagnostics are printed periodically from the capture and display tasks.

Capture diagnostics include:

- Frame period.
- I2S read/wait time.
- Record queue time.
- FFT time.
- Publication time for FFT, analyser, and oscilloscope data.
- Capture active duty.
- PCM min, max, mean, RMS, clipping.
- FFT average, peak, display noise floor, and whether the display noise floor
  calibration has completed.

SD diagnostics include:

- SD status.
- Current file.
- Filled/free queue depth.
- Chunks and bytes written.
- Current and cumulative queue drops.
- Queue high-water mark.
- Error count.
- Average and maximum SD write time.

Display diagnostics include:

- Display frame period.
- Wait time.
- Clear/frame/analyser/FFT/scope render time.
- Display active duty.
- Notification count.
- Timeout count.
- Display-frame drops.

## Current Known Edges

- FTP download while recording has been tested successfully.
- FTP upload/delete while recording should be treated as higher risk until tested.
- FTP and recording share the SD card, so SD write latency can rise during FTP
  transfers. The PSRAM queue is expected to absorb normal SD stalls.
- FTP transfer diagnostics are now direction-aware for upload/download start,
  progress, finish, and error events.
- The I2S LRCLK line was confirmed by continuity test to be wired to GPIO 40,
  not GPIO 42. Using GPIO 42 caused the microphone LRCLK input to float/pick up
  crosstalk, producing noisy capture that changed when the wiring was touched.
